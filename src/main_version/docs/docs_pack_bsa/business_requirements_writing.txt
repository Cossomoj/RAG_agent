## Основы бизнес-требований

### Определение и характеристики

**Бизнес-требования** — это высокоуровневые потребности организации, описывающие желаемые бизнес-результаты и их важность для достижения стратегических целей.

#### Ключевые характеристики качественных бизнес-требований:
1. **Бизнес-ориентированность**: Фокус на результатах, а не технологиях
2. **Измеримость**: Четкие KPI и критерии успеха
3. **Достижимость**: Реалистичные цели и сроки
4. **Согласованность**: Одобрение всех стейкхолдеров
5. **Трассируемость**: Связь со стратегией организации

### Типы бизнес-требований с примерами

#### 1. Стратегические требования
**Определение**: Долгосрочные цели организации (3-5 лет)

**Пример**:
```
BR-001: Цифровая трансформация розничных продаж
Цель: Перевести 70% продаж в онлайн-каналы к 2027 году
Обоснование: Изменение покупательского поведения, снижение издержек на содержание офлайн-точек
Метрики успеха: 
- Доля онлайн-продаж: 70%
- Снижение операционных расходов: 30%
- NPS онлайн-канала: >60
```

#### 2. Тактические требования
**Определение**: Среднесрочные цели подразделений (1-2 года)

**Пример**:
```
BR-002: Автоматизация обработки заявок
Цель: Сократить время обработки клиентских заявок с 3 дней до 4 часов
Обоснование: Повышение удовлетворенности клиентов, конкурентное преимущество
Метрики успеха:
- Среднее время обработки: <4 часа
- Процент автоматически обработанных заявок: >80%
- Удовлетворенность клиентов: рост на 25%
```

#### 3. Операционные требования
**Определение**: Краткосрочные улучшения процессов (3-6 месяцев)

**Пример**:
```
BR-003: Оптимизация процесса согласования документов
Цель: Внедрить электронное согласование для всех типов документов
Обоснование: Удаленная работа сотрудников, ускорение процессов
Метрики успеха:
- Охват процессов: 100%
- Сокращение времени согласования: 60%
- Экономия на бумаге и логистике: 500К руб/год
```

#### 4. Регулятивные требования
**Определение**: Требования законодательства и стандартов

**Пример**:
```
BR-004: Соответствие требованиям 152-ФЗ
Цель: Обеспечить защиту персональных данных согласно законодательству
Обоснование: Избежание штрафов, репутационные риски
Метрики успеха:
- Прохождение аудита: 100% соответствие
- Инциденты с ПД: 0
- Обученные сотрудники: 100%
```

### Роль аналитика в работе с бизнес-требованиями

#### Ключевые активности:
1. **Выявление** скрытых потребностей через интервью и анализ
2. **Формализация** неструктурированных идей в четкие требования
3. **Валидация** требований с бизнесом и техническими экспертами
4. **Приоритизация** на основе ценности и сложности
5. **Управление изменениями** в процессе реализации

#### Измеримые KPI аналитика:
- Время на формулирование требования: <2 дней
- Процент принятых с первого раза: >85%
- Количество изменений после утверждения: <15%
- Удовлетворенность стейкхолдеров: >4/5

### Практическое задание: Идентификация типов требований

### Анализ бизнес-контекста

#### Фреймворк 5W+1H для контекста:
1. **What** (Что?): Какую проблему решаем?
2. **Why** (Почему?): Почему это важно сейчас?
3. **Who** (Кто?): Кто заинтересован и вовлечен?
4. **When** (Когда?): Какие временные рамки?
5. **Where** (Где?): География и каналы?
6. **How** (Как?): Какие ресурсы и подходы?

#### Практический пример анализа контекста:
```markdown
**Проект**: Внедрение чат-бота для поддержки

**What**: Автоматизация первой линии поддержки клиентов
**Why**: 
- 70% обращений - типовые вопросы
- Поддержка работает только в будни
- Операторы перегружены (150 обращений/день)

**Who**:
- Клиенты: 50K активных
- Операторы поддержки: 15 человек
- IT департамент: 3 человека на проект

**When**: 
- Старт: Q1 2024
- MVP: Q2 2024
- Полный запуск: Q3 2024

**Where**:
- Веб-сайт компании
- Мобильное приложение
- Мессенджеры (Telegram, WhatsApp)

**How**:
- Бюджет: 5М рублей
- Технология: NLP на базе GPT
- Команда: Внутренняя + вендор
```

### Выявление проблем и возможностей

#### Техника "5 Почему" (5 Whys):
```
Проблема: Клиенты жалуются на долгое оформление заказов

Почему? → Процесс занимает 15 минут
Почему? → Нужно заполнить 20 полей
Почему? → Система требует все данные сразу
Почему? → Изначально проектировали для B2B
Почему? → Не учли потребности B2C сегмента

Корневая причина: Система не адаптирована под B2C
Решение: Упрощенный flow для физических лиц
```

#### Анализ AS-IS vs TO-BE:

**AS-IS (Текущее состояние)**:
```
[Клиент] → [Заполняет 20 полей] → [Ждет проверку] → [Оплачивает] → [Ждет подтверждение]
   ↓             15 мин                5 мин            2 мин            10 мин
                              Общее время: 32 минуты
```

**TO-BE (Желаемое состояние)**:
```
[Клиент] → [Quick checkout] → [Оплата] → [Instant подтверждение]
   ↓           3 мин           2 мин            0 мин
                    Общее время: 5 минут (-84%)
```

## Формулирование бизнес-целей

### Методология SMART для бизнес-целей

#### Разбор каждого критерия с примерами:

**S - Specific (Конкретная)**
❌ Плохо: "Улучшить продажи"
✅ Хорошо: "Увеличить онлайн-продажи в категории 'Электроника'"

**M - Measurable (Измеримая)**
❌ Плохо: "Повысить удовлетворенность"
✅ Хорошо: "Повысить NPS с 45 до 65 баллов"

**A - Achievable (Достижимая)**
❌ Плохо: "Захватить 90% рынка за месяц"
✅ Хорошо: "Увеличить долю рынка с 15% до 20% за год"

**R - Relevant (Релевантная)**
❌ Плохо: "Внедрить блокчейн" (если нет обоснования)
✅ Хорошо: "Сократить время верификации документов для соответствия регулятору"

**T - Time-bound (Ограниченная по времени)**
❌ Плохо: "В будущем"
✅ Хорошо: "К концу Q2 2024"

### Структура бизнес-требования

#### Полный шаблон с примером:
```markdown
**ID**: BR-2024-001
**Версия**: 1.2
**Дата**: 15.01.2024
**Автор**: Иванов И.И.
**Статус**: Утверждено

**Название**: Цифровизация процесса выдачи кредитов

**Краткое описание**: 
Автоматизировать процесс рассмотрения кредитных заявок для сокращения времени принятия решения и повышения качества оценки рисков.

**Бизнес-цель SMART**:
Сократить время принятия решения по кредитной заявке с 3 дней до 15 минут для 80% заявок физических лиц до 5М рублей к 01.07.2024.

**Бизнес-обоснование**:
- Рынок: Конкуренты принимают решения за 30 минут
- Клиенты: 40% отказов из-за долгого ожидания
- Экономика: Потери 50М руб/год из-за ухода клиентов
- Возможности: Рост портфеля на 30% при быстрых решениях

**Стейкхолдеры**:
- Спонсор: Директор розничного бизнеса
- Владелец: Начальник кредитного департамента  
- Пользователи: Кредитные специалисты (50 чел)
- Бенефициары: Клиенты банка

**Критерии успеха**:
1. Время решения: ≤15 минут для 80% заявок
2. Качество решений: Bad rate <5%
3. Конверсия: Рост одобрений на 20%
4. NPS процесса: >70 баллов
5. ROI: >250% за первый год

**Ограничения**:
- Бюджет: 25М рублей
- Срок: 6 месяцев
- Технологии: Использование существующей инфраструктуры
- Регулятор: Соответствие требованиям ЦБ РФ

**Предположения**:
- Доступность данных из бюро кредитных историй
- Готовность клиентов к цифровому процессу
- Стабильность регулятивных требований

**Риски**:
- Изменение требований регулятора (вероятность: средняя)
- Сопротивление сотрудников (вероятность: высокая)
- Технические сложности интеграции (вероятность: средняя)

**Связанные требования**:
- BR-2023-045: Единая система скоринга
- BR-2024-002: Мобильное приложение банка
- TR-2024-001: Интеграция с НБКИ

**История изменений**:
- v1.0 (10.01.2024): Первая версия
- v1.1 (12.01.2024): Уточнены критерии успеха
- v1.2 (15.01.2024): Добавлены риски
```

### Приоритизация бизнес-требований

#### Метод WSJF (Weighted Shortest Job First):
```
WSJF = (Business Value + Time Criticality + Risk Reduction) / Job Size
```

**Пример расчета**:
| Требование | Бизнес-ценность | Критичность времени | Снижение рисков | Размер работы | WSJF |
|------------|-----------------|---------------------|-----------------|---------------|------|
| Чат-бот | 8 | 5 | 3 | 5 | 3.2 |
| Кредитный скоринг | 9 | 9 | 8 | 8 | 3.25 |
| Мобильное приложение | 7 | 6 | 2 | 3 | 5.0 |

**Вывод**: Приоритет - Мобильное приложение (наибольший WSJF)

#### Матрица Value/Effort:
```
    Высокая ценность
           ↑
    [Quick Wins]  |  [Major Projects]
    • Чат-бот     |  • Новая CRM
    • A/B тесты   |  • Редизайн сайта
    ──────────────────────────────────→
    [Fill Ins]    |  [Time Wasters]   Высокие
    • Мелкие      |  • Блокчейн       усилия
      улучшения   |    эксперименты
           ↓
    Низкая ценность
```

### Связь бизнес-требований с KPI

#### Каскадирование целей:
```
Стратегическая цель компании
"Рост выручки на 30%"
         ↓
Бизнес-требование департамента
"Увеличить конверсию сайта с 2% до 4%"
         ↓
Функциональные требования
"Упростить процесс checkout до 3 кликов"
         ↓
Технические требования  
"Внедрить one-click purchase через Apple Pay"
```

#### Дерево KPI:
```
ROI проекта >200%
├── Доходы +50М руб/год
│   ├── Новые клиенты: +5000
│   ├── Средний чек: +15%
│   └── Частота покупок: +20%
└── Экономия 20М руб/год
    ├── Автоматизация: -10 FTE
    ├── Снижение ошибок: -70%
    └── Ускорение процессов: -60%
```

### Business Case: детальная разработка

#### Финансовый анализ:

**1. ROI Calculation**:
```
ROI = (Gain from Investment - Cost of Investment) / Cost of Investment × 100%

Пример:
- Инвестиции: 10М руб
- Доп. доход: 25М руб/год
- Экономия: 5М руб/год
- Период: 3 года

ROI = ((25+5)×3 - 10) / 10 × 100% = 800%
```

**2. NPV (Net Present Value)**:
```
NPV = Σ (Cash Flow / (1 + r)^t) - Initial Investment

где:
- r = ставка дисконтирования (15%)
- t = период (год)

Year 0: -10M
Year 1: +30M / 1.15 = 26.1M
Year 2: +30M / 1.32 = 22.7M
Year 3: +30M / 1.52 = 19.7M

NPV = -10 + 26.1 + 22.7 + 19.7 = 58.5M руб
```

**3. Payback Period**:
```
Накопленный cash flow:
Month 0: -10M
Month 4: -0M (Breakeven)
Month 12: +20M

Payback Period = 4 месяца
```

### Кейс 2: Конфликт требований стейкхолдеров

**Ситуация**: Разные видения мобильного банка

**Конфликтующие требования**:
1. **Marketing**: "Максимум функций и геймификация"
2. **Security**: "Минимум данных и сложная аутентификация"
3. **UX**: "Простота и минимализм"
4. **Compliance**: "Полное соответствие 115-ФЗ"

**Процесс разрешения**:

1. **Mapping интересов**:
```
Стейкхолдер | Ключевой интерес | Критичность
Marketing   | Вовлечение       | Важно
Security    | Защита данных    | Критично
UX          | Удобство         | Важно
Compliance  | Соответствие     | Критично
```

2. **Поиск win-win**:
- Биометрия вместо сложных паролей (Security + UX)
- Персонализация без сбора лишних данных (Marketing + Security)
- Гибкие настройки приватности (All)

3. **Приоритизация по MoSCoW**:
- **Must**: Безопасность, Compliance
- **Should**: Базовое удобство
- **Could**: Дополнительные функции
- **Won't**: Геймификация (в MVP)

**Результат**: Консенсус достигнут, MVP определен

## Заключение и best practices

### Топ-10 советов для работы с бизнес-требованиями:

1. **Начинайте с "Почему?"** - всегда понимайте бизнес-драйверы
2. **Говорите на языке бизнеса** - избегайте технического жаргона
3. **Visualize everything** - картинка стоит тысячи слов
4. **Validate early and often** - чем раньше найдете ошибку, тем дешевле
5. **Keep it SMART** - конкретные, измеримые цели
6. **Document decisions** - не только что, но и почему
7. **Manage stakeholders** - 50% успеха в коммуникации
8. **Think ROI** - всегда считайте бизнес-выгоду
9. **Embrace change** - изменения неизбежны, планируйте их
10. **Learn continuously** - каждый проект - это опыт

> "Хорошее бизнес-требование - это мост между мечтой бизнеса и реальностью технологий" - Мудрость опытных аналитиков
